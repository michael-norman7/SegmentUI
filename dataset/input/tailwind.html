Upgrade to Tailwind CSS v4: A Comprehensive Guide

Everything you need to know to smoothly transition your projects to the latest Tailwind CSS version.

Published on: March 10, 2024 15 minute read

Introduction

Tailwind CSS v4 is here, bringing performance improvements and exciting new features. While the upgrade is designed to be smooth, some breaking changes require attention. This guide provides a clear path to upgrade your projects from v3 to v4, ensuring a seamless transition and highlighting key improvements.

Key Takeaways

  • Tailwind CSS v4 offers performance enhancements and new features.
  • An automated upgrade tool is available for most projects.
  • Manual upgrade steps are outlined for specific configurations.
  • Several breaking changes require code adjustments.
  • Understanding the changes ensures a smooth upgrade process.

Using the Upgrade Tool

For most projects, the Tailwind CSS team has provided an automated upgrade tool. This tool handles the majority of the migration process, including dependency updates, configuration file migration to CSS, and template file adjustments.

npx @tailwindcss/upgrade

Before running the tool, ensure you have Node.js 20 or higher. It's highly recommended to run the upgrade tool in a new branch to review changes and test thoroughly before merging.

Upgrade Tool Highlights

  • Automates most of the upgrade process from v3 to v4.
  • Handles dependency updates and configuration migration.
  • Saves significant time and effort in upgrading projects.
  • Requires Node.js 20 or higher.
  • Always run in a new branch for safety and review.

Upgrading Manually

While the upgrade tool is recommended, understanding manual steps is crucial for complex projects or specific setups. Here's a breakdown of manual upgrade considerations:

Using PostCSS

In v4, the PostCSS plugin is now in @tailwindcss/postcss. Remove postcss-import and autoprefixer as they are now handled automatically.


export default {
  plugins: {
    "postcss-import": {}, // REMOVE
    tailwindcss: {},
    autoprefixer: {}, // REMOVE
    "@tailwindcss/postcss": {},
  },
};

Using Vite

Migrate to the dedicated Vite plugin @tailwindcss/vite for improved performance.


import { defineConfig } from "vite";
import tailwindcss from "@tailwindcss/vite";
export default defineConfig({
  plugins: [
    tailwindcss(),
  ],
});

Using Tailwind CLI

Tailwind CLI is now in @tailwindcss/cli. Update your build commands accordingly.


npx tailwindcss -i input.css -o output.css // OLD
npx @tailwindcss/cli -i input.css -o output.css // NEW

Manual Upgrade Tips

  • Understand PostCSS, Vite, and CLI specific changes.
  • Update dependencies to the new packages.
  • Review configuration files for necessary adjustments.
  • Test build commands after manual updates.

Key Changes from v3 to v4

Tailwind CSS v4 introduces several breaking changes. Understanding these is crucial for a successful upgrade. Here's a detailed list:

Removed @tailwind directives

Replace @tailwind directives with standard CSS @import "tailwindcss";.


@tailwind base;     // REMOVE
@tailwind components; // REMOVE
@tailwind utilities;  // REMOVE
@import "tailwindcss"; // ADD

Removed Deprecated Utilities

Several deprecated utilities from v3 have been removed. Use the modern alternatives:

  • bg-opacity-* replaced by bg-black/50
  • text-opacity-* replaced by text-black/50
  • border-opacity-* replaced by border-black/50
  • ... (and others listed in the original text)

Renamed Utilities

Some utilities have been renamed for consistency:

  • shadow-sm renamed to shadow-xs
  • shadow renamed to shadow-sm
  • ... (and others listed in the original text)

Updated shadow, radius, and blur scales

Scales have been updated to ensure every utility has a named value. Update utilities accordingly (e.g., shadow-sm to shadow-xs).

Renamed outline utility

outline-none is now outline-hidden. outline-none now sets outline-style: none.


 // OLD (was outline-hidden behavior)
 // NEW (outline-hidden behavior)
 // NEW (actual outline-none)

Default ring width change

The default ring utility is now 1px (was 3px). Use ring-3 to get the previous 3px ring.


 // OLD (3px ring)
 // NEW (3px ring)

Space-between selector change

Selector for space-x-* and space-y-* utilities changed for performance. May affect spacing with inline elements or custom margins.


/* Before */
.space-y-4 > :not([hidden]) ~ :not([hidden]) { /* ... */ }
/* Now */
.space-y-4 > :not(:last-child) { /* ... */ }

Variants with gradients

Variant gradients now preserve values. Use via-none to "unset" gradients in specific states.

Container configuration

Container utility customization is now done via @utility directive instead of configuration options.


@utility container {
  margin-inline: auto;
  padding-inline: 2rem;
}

Default border color

Default border color changed from gray-200 to currentColor. Ensure to specify border colors explicitly.

Default ring width and color

Default ring width is 1px, and color is currentColor. Use ring-3 and specify color if needed.

Preflight changes

  • New default placeholder color (current text color at 50% opacity).
  • Buttons use cursor: default.
  • Dialog margins removed.

Using a prefix

Prefixes now look like variants (e.g., tw:flex).

Adding custom utilities

Use @utility API instead of @layer utilities for custom utilities.


@utility tab-4 {
  tab-size: 4;
}

Variant stacking order

Variant stacking order is now left-to-right (was right-to-left).

Variables in arbitrary values

Use parentheses for CSS variables in arbitrary values: bg-(--brand-color) (was bg-[--brand-color]).

Hover styles on mobile

Hover variant now only applies on devices that support hover. Can override with @custom-variant hover (&:hover); if needed.

Transitioning outline-color

transition and transition-color now include outline-color.

Disabling core plugins

corePlugins option is no longer supported.

Using the theme() function

Prefer CSS variables over theme() function. Use CSS variable names in theme() (e.g., theme(--breakpoint-xl)).


@media (width >= theme(screens.xl)) { /* ... */ } // OLD
@media (width >= theme(--breakpoint-xl)) { /* ... */ } // NEW

Using a JavaScript config file

JavaScript config files are supported but not auto-detected. Use @config directive to load them.


@config "../../tailwind.config.js";

Theme values in JavaScript

resolveConfig removed. Use CSS variables directly in JavaScript.

Using @apply with Vue, Svelte, or CSS modules

Use @reference to import theme variables in separate CSS files or use CSS variables directly.

Key Changes Summary

  • @tailwind directives replaced with @import "tailwindcss".
  • Deprecated utilities removed; use modern alternatives.
  • Utilities renamed (e.g., shadow-sm to shadow-xs).
  • Outline and ring utilities behavior adjustments.
  • Configuration and customization method changes.
  • JavaScript and CSS variable usage updates.
